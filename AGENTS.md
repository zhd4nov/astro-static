# AGENTS.md

Базовые правила для ИИ-агентов в этом репозитории.

## Назначение

Этот файл задает постоянные принципы работы. Не добавляй сюда одноразовые задачи, текущие планы и временные решения.

Операционный порядок выполнения задач смотри в `docs/agent-workflow.md`. Внутренние роли одного агента описаны в `docs/agent-roles.md`.

## Приоритеты

1. Сохраняй смысл проекта.
2. Делай минимальное достаточное изменение.
3. Предпочитай ясность эффектности.
4. Фиксируй значимые решения письменно.
5. Проверяй результат доступным способом.

## Режимы

1. Продуктовый режим выполняет задачи сайта и не меняет правила системы.
2. Сервисный режим `РЕМОНТ` улучшает правила системы и не выполняет задачи сайта.
3. Запускай `РЕМОНТ` только по явному слову `РЕМОНТ` в запросе пользователя.
4. Без режима `РЕМОНТ` запрещено менять `AGENTS.md`, `docs/agent-workflow.md`, `docs/agent-roles.md` и защищенные системные документы.
5. Перечень защищенных системных документов указан в `docs/process-improvement.md`.

## Инварианты

1. Не откатывай чужие изменения без явного запроса.
2. Не удаляй и не переименовывай файлы без необходимости.
3. Не смешивай несвязанные задачи в одном изменении.
4. Не добавляй сложность без обоснования.
5. Не выдавай черновик за утвержденный материал.
6. Не скрывай невыполненную проверку.
7. Не расширяй скоуп без запроса пользователя.

## Продуктовые правила

1. Сохраняй назначение сайта понятным.
2. Не добавляй маркетинговое давление без явного требования проекта.
3. Ставь ясность, проверяемость и навигацию выше эффектности.
4. Помогай пользователю понять, подходит ли ему содержимое сайта.
5. Показывай границы материалов, услуг или возможностей явно.
6. Убирай текст, который не помогает пользователю принять следующий осознанный шаг.

## Язык и контент

1. Пиши документы на русском языке.
2. Используй английский только для терминов, имен собственных, путей, команд, кода и общепринятых профессиональных обозначений.
3. Если у термина есть естественный русский эквивалент, используй русский эквивалент.
4. Пиши спокойно, точно и предметно.
5. Избегай универсальных маркетинговых формул.
6. Помечай неподтвержденный текст как черновик.

## Архитектура

1. Фиксируй значимые технические решения письменно.
2. Не выбирай более сложный стек ради возможных будущих сценариев.
3. Предпочитай решения, которые агенту легко читать, менять и проверять.
4. Предпочитай хранение контента в простом текстовом формате.
5. Добавляй зависимости только после проверки пользы и стоимости поддержки.

## Документация

1. Обновляй `README.md`, если меняются команды, структура проекта или способ проверки.
2. Обновляй проектные документы, если меняются аудитории, страницы, цели или границы.
3. Не дублируй одно и то же правило в нескольких документах.
4. Ссылайся на источник правила вместо копирования текста.
5. Удаляй устаревшие инструкции вместе с заменой.

## Проверка

1. Перед изменениями проверяй состояние репозитория.
2. После изменений проверяй затронутые файлы.
3. Если есть команды проверки, запускай их.
4. Если проверки нет, укажи это в итоговом сообщении.
5. Если проверка невозможна, укажи причину.

## Итоговый ответ

1. Укажи, что изменено.
2. Укажи, где это находится.
3. Укажи, как проверено.
4. Укажи ближайший разумный следующий шаг, если он следует из задачи.
