# Инженерные стандарты

## Назначение

Этот документ задает стек-независимые правила разработки.

Стековые правила добавляй отдельным документом в `docs/engineering/` после утверждения стека.

## Базовые правила

1. Реализуй только поставленную задачу.
2. Соблюдай принятые ADR.
3. Держи изменения локальными.
4. Предпочитай простое решение сложному.
5. Не добавляй зависимость без обоснования.
6. Удаляй мертвый код.
7. Не оставляй неиспользуемые зависимости.
8. Не смешивай рефакторинг и новую функциональность без необходимости.
9. Обновляй документацию вместе с командами, структурой и поведением.
10. Делай результат воспроизводимым через команды проекта.

## Код

1. Пиши код с явными именами.
2. Разделяй ответственность модулей.
3. Не прячь побочные эффекты.
4. Обрабатывай ошибки явно.
5. Не добавляй глобальное состояние без причины.
6. Не усложняй абстракции до появления повторяющейся потребности.

## Типы и контракты

1. Используй строгую типизацию.
2. Описывай границы данных явно.
3. Валидируй внешние входы.
4. Не передавай неструктурированные данные между слоями.
5. Не отключай проверки типов без явной причины.

## Интерфейс

1. Соблюдай доступность.
2. Не ломай адаптивность.
3. Минимизируй клиентский код.
4. Проверяй основной пользовательский сценарий.
5. Не добавляй визуальные эффекты, если они мешают смыслу или скорости.

## Зависимости

1. Добавляй зависимость только с причиной.
2. Не добавляй зависимость ради одной простой функции.
3. Удаляй зависимость, если она больше не используется.
4. Указывай новые зависимости в итоговом сообщении.
5. Выноси спорную зависимость в ADR.

## Тесты

1. Покрывай тестами изменяемую логику.
2. Добавляй регрессионный тест при исправлении дефекта.
3. Проверяй маршруты и рендеринг, если изменение влияет на страницы.
4. Проверяй пользовательский сценарий при изменении интерфейса.
5. Не подменяй автоматическую проверку ручной без причины.

## После утверждения стека

После утверждения стека добавь стековые правила в `docs/engineering/`.

Стековые правила должны фиксировать:

1. Правила структуры проекта.
2. Правила типизации.
3. Правила модулей и границ.
4. Правила тестирования.
5. Правила для монорепозитория, если используется монорепозиторий.

Если используется монорепозиторий, добавь корневые команды для проверки всего репозитория и отдельных частей.
