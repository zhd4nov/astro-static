# Роли агента

## Назначение

Этот документ описывает внутренние фокусы одного агента.

Роли не создают отдельных исполнителей, отдельных контекстов и обязательных артефактов. Агент переключает фокус по ходу задачи и использует только те артефакты, для которых есть основание.

## Постановка

Фокус постановки нужен, чтобы понять запрос и сохранить намерение пользователя.

Агент:

1. Отделяет цель, границы, ограничения и критерии готовности.
2. Фиксирует блокирующие неизвестные.
3. Выбирает минимальный уровень задачи по `docs/agent-workflow.md`.
4. Создает `TASK` только для обычной или сложной задачи.

## Проектирование

Фокус проектирования нужен, если задача требует выбора подхода, нового поведения или согласования нескольких частей.

Агент:

1. Проверяет существующую архитектуру и документы.
2. Создает `ADR` только при архитектурном выборе.
3. Создает `SPEC` только когда без нее нельзя безопасно реализовать изменение.
4. Не добавляет сложность ради возможных будущих сценариев.

## Реализация

Фокус реализации нужен для внесения минимального достаточного изменения.

Агент:

1. Работает в границах принятой задачи.
2. Сохраняет чужие изменения.
3. Не меняет правила системы во время продуктовой задачи.
4. Обновляет только документы, которые действительно затронуты изменением.

## Проверка

Фокус проверки нужен, чтобы подтвердить результат доступным способом.

Агент:

1. Определяет затронутые файлы и применимые команды.
2. Запускает доступные проверки.
3. Проверяет пользовательский сценарий, если он есть.
4. Прямо указывает невыполненные проверки и причину.

## Закрытие

Фокус закрытия нужен, чтобы завершить задачу без расширения скоупа.

Агент:

1. Сверяет результат с целью и критериями готовности.
2. Фиксирует выполненные и невыполненные проверки.
3. Обновляет `TASK` и `docs/tasks/index.md`, если задача создавалась.
4. Не добавляет новую работу в закрываемую задачу.

## РЕМОНТ

Фокус `РЕМОНТ` нужен только для изменения правил системы.

Агент:

1. Запускает этот фокус только по явному слову `РЕМОНТ`.
2. Не выполняет продуктовую задачу одновременно с ремонтом правил.
3. Меняет системные документы только при явном основании.
4. Сохраняет правила короткими, проверяемыми и непротиворечивыми.
