# Рабочий маршрут агента

## Назначение

Этот документ задает основной маршрут выполнения задач. Постоянные правила находятся в `AGENTS.md`, роли одного агента - в `docs/agent-roles.md`.

## Маршрут

1. Прими запрос и сохрани намерение пользователя.
2. Проверь состояние репозитория через `git status --short`.
3. Прочитай минимальный контекст: `AGENTS.md`, этот документ и только нужные проектные файлы.
4. Выбери минимальный уровень задачи.
5. Выполни изменение в границах запроса.
6. Проверь результат доступным способом по `docs/quality-gates.md`.
7. Обнови только документы, которые действительно затронуты.
8. Кратко отчитайся: что изменено, где находится, как проверено, какой следующий шаг нужен.

## Уровни задач

### Мелкая

Используй для локальной правки с ясной целью, малым риском и понятной проверкой.

Правила:

1. `TASK` не нужен.
2. `ADR`, `SPEC`, `REVIEW` и `RUN` не создаются.
3. Достаточно итогового сообщения с проверками.

### Обычная

Используй для значимого изменения с понятными границами, которое нужно отследить.

Правила:

1. Создай один `TASK` по `docs/task-management.md`.
2. Не создавай `ADR`, `SPEC`, `REVIEW` или `RUN` без отдельного основания.
3. Обновляй `docs/tasks/index.md` при изменении статуса или связанных артефактов.

### Сложная

Используй для изменения с архитектурным выбором, несколькими связанными зонами, спорной приемкой или высоким риском.

Правила:

1. Создай `TASK`.
2. Создай `ADR` только при архитектурном выборе.
3. Создай `SPEC` только если без нее нельзя безопасно согласовать реализацию.
4. Создай файловое ревью только по основаниям из `docs/review-standards.md`.
5. Создай `RUN` только для ретроспективы, сервисного режима или явного запроса.

## Постановка

1. Сохрани исходное намерение пользователя.
2. Отдели цель, границы, ограничения и критерии готовности.
3. Зафиксируй неизвестное.
4. Задавай вопросы только при блокирующей неоднозначности.
5. Выбери минимальный уровень задачи.

Подробный порядок постановки смотри в `docs/task-intake.md`.

## Реализация

1. Работай с минимальным достаточным набором файлов.
2. Не откатывай чужие изменения.
3. Не расширяй скоуп без запроса пользователя.
4. Следуй существующим паттернам проекта.
5. Фиксируй значимые решения через `ADR`, если для них есть основание.

## Проверка

1. Определи затронутые файлы.
2. Определи доступные команды из `README.md` и `Makefile`.
3. Запусти применимые проверки.
4. Проверь пользовательский сценарий, если он есть.
5. Укажи невыполненные проверки и причину.

Подробные проверки смотри в `docs/quality-gates.md`.

## Документы

Обновляй документы только когда изменение действительно меняет соответствующий предмет:

- `README.md` - команды, структура проекта или способ проверки.
- Проектные документы - аудитории, страницы, цели или границы.
- `TASK` и `docs/tasks/index.md` - отслеживаемые обычные и сложные задачи.
- `ADR` - архитектурные решения.
- `SPEC` - согласование сложной реализации.
- `REVIEW` - файловая приемка по основаниям из `docs/review-standards.md`.
- `RUN` - ретроспектива или сервисный анализ.

## РЕМОНТ

Режим `РЕМОНТ` отделен от продуктовой работы.

1. Запускай его только по явному слову `РЕМОНТ`.
2. Используй правила `docs/process-improvement.md`.
3. Меняй системные документы только в границах запроса.
4. После изменения правил проверь, что документы не пересказывают друг друга.
