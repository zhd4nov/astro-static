# Рабочий процесс агента

## Назначение

Этот документ описывает повторяемый порядок выполнения задач. Он не должен содержать одноразовых планов, текущих поручений или решений, которые устареют после следующей задачи.

Базовые правила проекта находятся в `AGENTS.md`.

Улучшение правил системы выполняется только в режиме `РЕМОНТ` по `docs/process-improvement.md`.

## Цикл работы

1. Прими запрос пользователя.
2. Определи, является ли запрос мелкой правкой или значимым изменением.
3. Для значимого изменения создай или используй задачу из `docs/tasks/`.
4. Проверь состояние репозитория.
5. Прочитай минимально необходимый контекст.
6. Определи границы изменения.
7. Выполни изменение.
8. Проверь результат.
9. Сообщи итог кратко.

## Чтение контекста

Всегда читай:

1. `AGENTS.md`.
2. `README.md`.

Читай по необходимости:

- `docs/process-improvement.md` - только для режима `РЕМОНТ`.
- `docs/process-runs/` - для ретроспективы, журналов процесса и режима `РЕМОНТ`.
- `docs/task-intake.md` - для преобразования запроса пользователя в задачу.
- `docs/task-closeout.md` - для закрытия принятой задачи.
- `docs/task-management.md` - для идентификации и трассируемости задач.
- `docs/sdlc-limits.md` - для лимитов проходов, блокировки и эскалации.
- `docs/tasks/` - для входной задачи.
- `docs/adr/` - для задач про стек, архитектуру, зависимости, хранение данных, деплой и интеграции.
- `docs/specs/` - для реализации по спецификации.
- `docs/engineering-standards.md` - для задач с кодом.
- `docs/quality-gates.md` - для проверки результата.
- Проектные документы нового сайта - если они уже созданы в проекте на основе template.

Не перечитывай все документы без причины. Выбирай контекст по задаче.

## Режимы

Рабочий режим:

1. Выполняй задачи сайта.
2. Не меняй `AGENTS.md`, `agents/` и защищенные системные документы.
3. Если нужно улучшить правила, зафиксируй кандидат в итоге задачи или итоговом сообщении.
4. Создавай журнал процесса только если он нужен для ретроспективы, разбора сложного случая или явно запрошен.

Режим `РЕМОНТ`:

1. Запускай только по явному слову `РЕМОНТ`.
2. Используй `agents/process-improvement-agent.md`.
3. Анализируй зафиксированные артефакты и обнаруженные противоречия.
4. Не выполняй задачи сайта.

## Определение границ

Перед правкой сформулируй для себя:

1. Какую проблему решает задача.
2. Какие файлы нужно изменить.
3. Какие файлы нельзя трогать.
4. Какая проверка подтвердит результат.

Если задача неоднозначна, выбери самый малый обратимый шаг. Задавай вопрос только при риске дорогого или необратимого решения.

## Размер изменения

Делай изменение настолько малым, насколько позволяет задача.

Не смешивай:

- выбор технологии и реализацию интерфейса;
- структуру сайта и тексты всех страниц;
- новый функционал и крупный рефакторинг;
- документацию процесса и требования сайта;
- временный план и постоянные правила.

## Артефакты

Мелкие правки можно выполнять без `TASK`, `SPEC`, `ADR` и `REVIEW`.

`TASK` обязателен только для значимых изменений по `docs/task-management.md`.

`ADR` нужен только при выборе архитектуры, стека, зависимости, хранения данных, деплоя или интеграции.

`SPEC` нужна для значимых изменений с несколькими файлами, новым поведением или проверяемыми критериями.

`REVIEW` нужен перед закрытием значимого изменения.

`RUN`-журналы не обязательны для каждой задачи. Используй их для ретроспективы, сервисного режима или сложного изменения, где полезно сохранить наблюдения процесса.

Если задача уже есть, не создавай новую без необходимости.

После каждого этапа значимой задачи обновляй файл задачи и `docs/tasks/index.md`.

Перед повторным проходом проверь лимиты в `docs/sdlc-limits.md`.

## Реализация

Если задача требует изменения кода, используй:

1. `agents/implementation-agent.md`.
2. `docs/engineering-standards.md`.
3. `docs/quality-gates.md`.

Не передавай реализационному агенту задачу с неразрешенным архитектурным решением.

Для значимой реализационной задачи используй спецификацию как основной вход.

Без спецификации выполняй только мелкую правку по определению из `docs/task-management.md`.

## Работа с контентом

1. Используй проектные документы нового сайта, если они уже созданы.
2. Если проектных документов нет, сохраняй нейтральный смысл template и не выдумывай цели сайта.
3. Пиши документы на русском языке.
4. Используй английский только для терминов, путей, команд, кода и общепринятых обозначений.
5. Помечай неподтвержденный текст как черновик.

## Проверка

Выполняй проверки по `docs/quality-gates.md`.

## Ревью

Значимое изменение допускается к закрытию только после ревью.

Для ревью используй:

1. `agents/review-agent.md`.
2. `docs/review-standards.md`.
3. `docs/reviews/template.md`.

Если ревью вернуло статус `Требуются изменения`, передай результат ревью реализационному агенту как вход для доработки.

Если ревью вернуло статус `Заблокировано`, вернись к архитектурному агенту или пользователю.

Если лимит ревью исчерпан, поставь задаче статус `Заблокирована`.

## Закрытие

После ревью со статусом `Принято` используй `agents/task-closeout-agent.md`.

Задача считается завершенной только после закрытия.

Закрытие должно обновить задачу, `docs/tasks/index.md` и затронутые проектные документы.

## Передача результата

В итоговом сообщении укажи:

1. Что изменено.
2. Где это находится.
3. Как проверено.
4. Что осталось нерешенным, если такое есть.

Не пересказывай документы целиком. Сообщай только информацию, нужную для продолжения работы.
