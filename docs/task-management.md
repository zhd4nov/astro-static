# Управление задачами

## Назначение

Этот документ является справочником по `TASK` и связанным артефактам. Уровни задач определены в `docs/agent-workflow.md`.

## Когда создавать TASK

`TASK` создается для обычной и сложной задачи.

`TASK` не нужен для мелкой задачи с ясной целью, малым риском и проверяемым результатом.

## Идентификаторы

Используй формат:

- `TASK-NNNN-short-name`
- `ADR-NNNN-short-name`
- `SPEC-NNNN-short-name`
- `REVIEW-NNNN-rX-short-name`
- `RUN-YYYY-MM-DD-short-name`

`NNNN` выбирай по следующему свободному номеру в соответствующей директории или реестре. `short-name` пиши латиницей, коротко и предметно.

## Статусы TASK

Используй только эти статусы:

- `Предложена`
- `Готова к реализации`
- `В реализации`
- `На ревью`
- `Принята`
- `Требуются изменения`
- `Заблокирована`
- `Заменена`

## Минимальное содержание TASK

В задаче должно быть:

1. Статус.
2. Исходный запрос.
3. Цель.
4. Границы.
5. Ограничения и допущения.
6. Критерии готовности.
7. Открытые вопросы или `нет`.
8. Связанные артефакты или `нет`.
9. Итог закрытия после завершения.

## Реестр задач

Обновляй `docs/tasks/index.md`, когда:

1. Создана новая `TASK`.
2. Изменился статус.
3. Добавлен или изменен связанный `ADR`, `SPEC` или `REVIEW`.
4. Задача принята, заблокирована или заменена.

## Связанные артефакты

Создавай дополнительные артефакты только по основанию:

- `ADR` - есть архитектурный выбор, новая зависимость, изменение стека, хранения данных, деплоя или интеграции.
- `SPEC` - есть сложная реализация, новое поведение в нескольких зонах или нужны проверяемые договоренности до кода.
- Файловое `REVIEW` - есть архитектурный риск, возврат на доработку, спорная приемка или явный запрос.
- `RUN` - нужен журнал ретроспективы, сервисного режима или пользователь явно попросил журнал.
