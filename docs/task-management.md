# Управление задачами

## Назначение

Этот документ задает правила идентификации и трассируемости задач.

Задача является входом в агентную систему после нормализации пользовательского запроса.

Правила нормализации находятся в `docs/task-intake.md`.

Лимиты проходов находятся в `docs/sdlc-limits.md`.

Правила закрытия находятся в `docs/task-closeout.md`.

## Единица работы

Каждая значимая работа получает `Task ID`.

Формат:

`TASK-NNNN-short-name`

Пример:

`TASK-0001-select-site-stack`

## Значимое изменение

Значимое изменение:

1. Меняет поведение.
2. Меняет пользовательский сценарий.
3. Меняет страницу, маршрут, компонент, модуль или интеграцию.
4. Меняет каркас проекта.
5. Меняет команды, проверки, зависимости или структуру.
6. Меняет смысл сайта, контентную структуру или границы проекта.
7. Требует критериев приемки.
8. Передается другому агенту.

Мелкая правка не меняет поведение, структуру, правила, команды и договоренности.

Мелкую правку можно выполнять без `TASK`, `SPEC`, `ADR` и `REVIEW`.

## Выдача ID

Перед созданием артефакта проверь существующие файлы и реестры.

Используй следующий свободный номер: `max(NNNN) + 1`.

Не переиспользуй номера удаленных, замененных или заблокированных артефактов.

Правило действует для `TASK`, `ADR`, `SPEC`, `REVIEW` и `RUN`, если соответствующий артефакт создается.

## Связанные артефакты

Используй общий номер, если связь один к одному:

- `TASK-0001-short-name`
- `ADR-0001-short-name`
- `SPEC-0001-short-name`
- `REVIEW-0001-short-name`

Если у задачи несколько ADR или ревью, каждый артефакт обязан ссылаться на `Task ID`.

## Когда создавать задачу

Создавай задачу для значимого изменения.

Не создавай задачу для мелкой правки, которая не меняет поведение, структуру, правила, команды и договоренности.

Создавай задачу только после обработки запроса по `docs/task-intake.md`.

## Где хранить задачи

Храни задачи в `docs/tasks/`.

Используй шаблон `docs/tasks/template.md`.

Веди общий реестр задач в `docs/tasks/index.md`.

Файл конкретной задачи является источником деталей.

Реестр задач является источником быстрого поиска статуса и связанных артефактов.

Обновляй реестр при:

1. Создании задачи.
2. Изменении статуса задачи.
3. Добавлении ADR.
4. Добавлении спецификации.
5. Добавлении ревью.
6. Изменении последнего ревью.
7. Закрытии задачи.

## Статусы

Используй один статус:

- `Предложена`;
- `Готова к реализации`;
- `В реализации`;
- `На ревью`;
- `Принята`;
- `Требуются изменения`;
- `Заблокирована`;
- `Заменена`.

## Трассируемость

Каждый ADR, спецификация и ревью должны ссылаться на `Task ID`.

Каждая задача должна ссылаться на:

1. Исходный запрос пользователя.
2. Связанные ADR.
3. Связанные спецификации.
4. Все связанные ревью.
5. Последнее ревью.
6. Текущий статус.
7. Следующего агента или пользователя.

После каждого этапа обновляй задачу:

1. После ADR добавь ссылку на ADR.
2. После ADR или спецификации поставь статус `Готова к реализации`, если задача готова к реализации.
3. После передачи в реализацию поставь статус `В реализации`.
4. После реализации поставь статус `На ревью`.
5. После ревью добавь ссылку на новое ревью, обнови последнее ревью и статус.
6. После доработки сохрани историю ревью и обнови последнее ревью.
7. После принятого ревью закрой задачу и обнови статус на `Принята`.

После каждого обновления задачи обновляй `docs/tasks/index.md`.

После каждого изменения статуса обновляй следующего агента в задаче и `docs/tasks/index.md`.

Если лимит из `docs/sdlc-limits.md` исчерпан, поставь статус `Заблокирована`.

Статус после ревью:

1. `Принято` в ревью - статус задачи `Принята` после закрытия.
2. `Требуются изменения` в ревью - статус задачи `Требуются изменения`.
3. `Заблокировано` в ревью - статус задачи `Заблокирована`.

## Передача между агентами

1. Агент постановки задачи получает сырой запрос пользователя.
2. Агент постановки задачи создает или выбирает задачу.
3. Архитектурный агент получает задачу как вход, если нужны ADR или SPEC.
4. Архитектурный агент создает ADR и спецификацию, если они нужны.
5. Реализационный агент получает задачу, спецификацию и связанные ADR, если они есть.
6. Ревью-агент получает задачу, спецификацию, связанные ADR, результат реализации и проверки.
7. Агент закрытия получает задачу и последнее принятое ревью.
8. При возврате на доработку реализационный агент получает последнее ревью.
