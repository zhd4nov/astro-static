# Постановка задачи

## Назначение

Этот документ задает правила обработки входящего запроса пользователя перед созданием `TASK`.

Цель этапа - получить четкую, проверяемую и ограниченную задачу для следующих агентов.

## Вход

Входом является сырой запрос пользователя.

Запрос может содержать:

1. Цель.
2. Контекст.
3. Ограничения.
4. Предпочтения.
5. Нерелевантные детали.
6. Разговорные формулировки.
7. Противоречия.

## Нормализация

Преобразуй запрос в задачу так:

1. Сохрани намерение пользователя.
2. Убери повторы и нерелевантные детали.
3. Замени абстрактные формулировки на проверяемые.
4. Раздели цель, контекст, ограничения и критерии готовности.
5. Зафиксируй, что входит в задачу.
6. Зафиксируй, что не входит в задачу.
7. Зафиксируй, что запрещено.
8. Сохрани неизвестное как открытый вопрос.

## Когда задавать вопросы

Задавай уточняющие вопросы только при блокирующей неоднозначности.

Не задавай вопросы, если:

1. Можно выбрать минимальный обратимый вариант.
2. Вопрос не влияет на следующий этап.
3. Ответ можно получить из документов проекта.
4. Неопределенность можно явно зафиксировать как ограничение.

## Качество формулировки

Формулировка задачи должна:

1. Быть конкретной.
2. Быть проверяемой.
3. Иметь явные границы.
4. Не содержать лишнего контекста.
5. Не содержать скрытых архитектурных решений.
6. Не содержать реализации, если она еще не выбрана.
7. Быть пригодной для передачи следующему агенту.

## Создание TASK

Создавай `TASK`, если запрос является значимым изменением.

Значимое изменение определяется в `docs/task-management.md`.

Используй шаблон `docs/tasks/template.md`.

После создания задачи обнови `docs/tasks/index.md`.

Перед повторным уточнением проверь лимит в `docs/sdlc-limits.md`.

Поставь статус:

1. `Готова к реализации`, если задача готова к реализации или следующему проектному шагу.
2. `Предложена`, если есть блокирующие вопросы.

## Следующий агент

Укажи следующий шаг:

1. `agents/architect-spec-agent.md` - если нужен ADR или спецификация.
2. `agents/implementation-agent.md` - если задача мелкая или спецификация уже есть.
3. `agents/review-agent.md` - если запрос касается приемки готовой работы.
4. Пользователь - если есть блокирующие вопросы.

## Запрещено

1. Не превращай TASK в пересказ запроса.
2. Не добавляй неподтвержденные требования.
3. Не смешивай постановку задачи с ADR.
4. Не смешивай постановку задачи со спецификацией.
5. Не смешивай постановку задачи с реализацией.
6. Не скрывай открытые вопросы.
