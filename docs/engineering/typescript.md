# TypeScript

## Назначение

Этот документ задает правила использования TypeScript в проекте.

Правила обязательны для любого кода на TypeScript.

## Базовый режим

1. Используй строгую типизацию.
2. Включай `strict`.
3. Включай `noImplicitAny`.
4. Включай `strictNullChecks`.
5. Включай `noUncheckedIndexedAccess`.
6. Включай `exactOptionalPropertyTypes`.
7. Не отключай проверки TypeScript локально.

## Запреты

1. Не используй `any`.
2. Не используй `as`.
3. Не используй приведение через `<Type>value`.
4. Не используй двойное приведение через `unknown`.
5. Не используй non-null assertion `!`.
6. Не используй `@ts-ignore`.
7. Не используй `@ts-expect-error`.
8. Не расширяй типы локальными обходами ради прохождения компиляции.
9. Не скрывай ошибку типов через промежуточные переменные.

Если без исключения невозможно продолжить, остановись и вынеси вопрос в ADR или спецификацию.

## Внешние данные

1. Считай внешние данные неизвестными до проверки.
2. Валидируй входные данные на границе системы.
3. Не передавай непроверенные внешние данные в доменный код.
4. После проверки используй явно описанный тип.
5. Не смешивай тип валидации и доменный тип без необходимости.

## Модели и контракты

1. Описывай публичные контракты явно.
2. Переиспользуй существующие типы.
3. Не дублируй тип, если он уже выражен в доменной модели.
4. Не создавай локальный тип, если нужен общий контракт.
5. Выноси общий тип рядом с владельцем данных.
6. Не экспортируй внутренний тип без необходимости.

## Выведение типов

1. Используй выведение типов для локальных очевидных значений.
2. Указывай тип явно на публичных границах.
3. Указывай тип результата у экспортируемых функций.
4. Указывай тип параметров у всех функций.
5. Не заставляй читателя восстанавливать контракт по реализации.

## Объекты и состояния

1. Используй дискриминируемые объединения для состояний.
2. Проверяй исчерпываемость ветвлений через `never`.
3. Не моделируй состояние набором несвязанных булевых флагов.
4. Используй `readonly` для данных, которые не должны изменяться.
5. Не мутируй входные параметры.

## Обобщения

1. Используй generic только при реальной связи между входом и выходом.
2. Не добавляй generic для одного конкретного типа.
3. Ограничивай generic через `extends`, если допустимы не все типы.
4. Не используй generic для обхода строгой типизации.

## Ошибки

1. Не бросай и не передавай нетипизированные ошибки через границы модулей.
2. Описывай ожидаемые ошибки явно.
3. Не используй строку как полноценный тип ошибки.
4. Обрабатывай неизвестную ошибку через явное сужение типа.

## Импорт и экспорт типов

1. Используй `import type` для типов.
2. Используй `export type` для типов.
3. Не смешивай runtime-импорт и type-импорт без необходимости.
4. Не создавай циклические зависимости между типами и реализацией.

## Тесты типов

1. Добавляй проверку типов в обязательные команды качества.
2. Проверяй публичные контракты через компиляцию.
3. Не считай прохождение тестов заменой проверке типов.
