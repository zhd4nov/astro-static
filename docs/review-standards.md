# Стандарты ревью

## Назначение

Этот документ задает правила приемки работы реализационного агента.

Ревью не заменяет проверки качества. Ревью принимает или возвращает результат после проверок.

Лимиты повторного ревью находятся в `docs/sdlc-limits.md`.

## Обязательная трассируемость

Каждое ревью должно ссылаться на:

1. Задачу.
2. Спецификацию.
3. Связанные ADR или `нет`.
4. Измененные файлы.
5. Выполненные проверки.
6. Критерии приемки.
7. Предыдущее ревью, если была доработка.

Без ссылки на спецификацию значимое изменение не принимается.

Без ссылки на задачу значимое изменение не принимается.

## Версионирование ревью

1. Каждый проход ревью сохраняй отдельным файлом.
2. Не перезаписывай предыдущий файл ревью.
3. Используй формат `REVIEW-NNNN-rX-short-name.md`.
4. Первый проход ревью обозначай `r1`.
5. Каждый следующий проход увеличивает номер ревью на 1.
6. Повторное ревью должно ссылаться на предыдущее ревью.
7. Последнее ревью со статусом `Принято` разрешает закрытие задачи.
8. Не создавай ревью после `r3` без нового решения пользователя или архитектурного агента.

## Проверка спецификации

1. Проверь, что реализация находится в границах спецификации.
2. Проверь, что запрещенные изменения не выполнены.
3. Проверь, что все критерии приемки закрыты.
4. Проверь, что открытые вопросы не блокируют приемку.

## Проверка качества

1. Проверь выполнение `docs/quality-gates.md`.
2. Проверь выполнение обязательной проверки типов.
3. Проверь тесты для измененной логики.
4. Проверь пользовательский сценарий для изменений интерфейса.
5. Проверь, что невыполненные проверки имеют причину.

## Проверка инженерных стандартов

1. Проверь соответствие `docs/engineering-standards.md`.
2. Проверь соответствие стековым правилам из `docs/engineering/`.
3. Проверь отсутствие обходов типизации.
4. Проверь обоснование новых зависимостей.
5. Проверь отсутствие незаявленного расширения объема.

## Замечания

Блокирующее замечание:

- нарушает спецификацию;
- нарушает ADR;
- нарушает обязательную проверку;
- нарушает строгую типизацию;
- ломает пользовательский сценарий;
- оставляет критерий приемки незакрытым.

Неблокирующее замечание:

- не мешает приемке;
- не нарушает спецификацию;
- может быть вынесено в отдельную задачу.

## Возврат на доработку

Если статус `Требуются изменения`, укажи:

1. Блокирующее замечание.
2. Ссылку на критерий приемки или правило.
3. Ожидаемое исправление.
4. Проверку, которая подтвердит исправление.

Реализационный агент обязан использовать последний результат ревью как вход для доработки.

Если одно блокирующее замечание повторяется в двух ревью подряд, останови возврат на доработку и зафиксируй причину.

## Приемка

Ставь статус `Принято`, только если:

1. Все блокирующие замечания закрыты.
2. Все критерии приемки выполнены.
3. Обязательные проверки выполнены или невозможность явно обоснована.
4. Документация обновлена.
5. Трассируемость полная.

Не принимай работу с проверкой в статусе `провалена`.

Не принимай работу с обязательной проверкой в статусе `не настроена` после утверждения стека.

После статуса `Принято` передай задачу агенту закрытия.
